# Doc-code coherence engine

Stardust is an indexer and frontmatter linter today; it must become a doc-code coherence engine, because the hard problem in repo doc management is not finding the doc, it is keeping docs true to the code as the code moves underneath them.

<details>
<summary><b>Problem</b></summary>
<br>

Docs in a repo rot silently. A spec written against `internal/store/daemon.go` stays byte-identical while the code it describes gains twelve commits, and nobody can tell from the doc that it is now archaeology. Generic notes tools cannot solve this because they do not live in the repo and cannot see the code move. Stardust does live in the repo and already reads git, so it is uniquely placed to bind docs to code and flag drift. The tool that makes repo docs trustworthy instead of stale-by-default wins; everything else (search, lint, scaffolding) is table stakes.

Today Stardust ships three concrete defects that block that thesis, verified against source:

1. The generator and validator disagree on the schema. `internal/service/docs.go:renderNewDoc` (lines 99-117) stamps `title, type, status, created, updated`; `internal/convention/check.go:checkDocFile` (line 103) demands those same five via a hardcoded `[]string{...}` loop; but the per-collection schema in `.stardust/collections/specs/config.toml` declares only `title` and `status`, and `internal/cli/init.go:docCollectionConfig` (lines 122-133) codegens only those two. Three sources of truth, three different field sets. A `collections.Validate(frontmatter, fields)` function already exists (`internal/collections/collections.go:130`) but the checker never calls it.

2. The graph is blind to its own edges. `internal/vault/vault.go:ExtractLinks` (lines 45-58) extracts wikilinks only. `internal/graph/graph.go:Build` (line 50) sets `Out: note.Links`, so `related:` frontmatter refs and inline code-path references never become edges. `internal/convention/check.go:checkRelated` (lines 124-140) validates that `related:` targets exist on disk but never feeds them to the graph. Result: a doc linked only by `related:` reports as an orphan (`graph.go:Orphans`, line 69, counts only `In`/`Out`), and `PersonalizedPageRank` bundle expansion (`bundle.go:59`) cannot reach it.

3. The differentiator is off. There is no `vectors` config field at all; vectors are a pure runtime capability gated on `s.embed.Available(ctx)` (`internal/service/index.go:51`). When Ollama is unreachable the run degrades to FTS-only and only an `IndexStats.Vectors bool` records it; the degrade happens mid-run (`index.go:79`) and is never surfaced loudly to the agent, which keeps consuming `Hybrid` as if semantic were live. The content-hash skip is note-level (`index.go:69`), so any one-character edit re-embeds every chunk of the note. The cross-encoder reranker exists (`internal/rerank/rerank.go`) but `RerankerURL` defaults to empty (`config.go:34`) and nothing wires it into `Query`. Semantic plus rerank is the only reason to choose this tool over ripgrep, and it is dark.

Drift detection, the moat, exists in embryo: `annotateStaleness` (`internal/service/governs.go:192`) and `checkStale` (`check.go:169`) compute commit-distance between a doc and the code it governs. But both are gated on a `governs:` glob plus `status: Implemented`, so the common case (an ADR that points at `daemon.go` through `related:` or an inline path, with no `governs:` block) is never checked for drift.

</details>

<details>
<summary><b>Context and background</b></summary>
<br>

Stardust is local-first, git-backed, markdown-truth (`CLAUDE.md`). Files are the source of truth; the SQLite index, link graph, and embeddings are disposable caches regenerated by `stardust rebuild`. Storage is deliberately boring (sqlite, FTS5, git, Ollama); the value is in routing and fusion. Constraints that shape every option below:

- Pure Go, single static binary. `modernc.org/sqlite` cannot load the sqlite-vec C extension, so vectors are brute-force cosine in Go (`internal/index/search.go`). No cgo, no C extensions.
- Content hash, never mtime, is the index authority, because cloud-synced storage rewrites mtimes (`CLAUDE.md`, `vault.ContentHash`).
- Git is already the change feed: `internal/gitx/gitx.go` exposes `DiffNames`, `LastCommit`, `CommitCountSince`, `HeadSHA`, `IsRepo`. The digest (`service/digest.go`) and incremental index (`service/index.go`) already key off it. Drift detection rides the same rail.
- Go conventions (`.claude/rules/go.md`): `%w` wrapping, never panic, `// --- Section ---` separators, doc comments on exports, no em or en dashes anywhere.

Prior decisions this work composes with, not against: ADR 0005 (Stardust manages docs in both repos), ADR 0006 (jrpc2 and shared conventions, `collections.ErrValidation` maps to the JSON-RPC domain band), ADR 0007 (compose hooks, never clobber). The drift surface must reach the agent through the same `--output json|md` boundary the existing service methods use.

No external research was required: every claim here is grounded in the current source tree, file and line cited.

</details>

<details>
<summary><b>Goals</b></summary>
<br>

1. One declarative schema per collection is the single source of truth that the scaffolder codegens from, the checker validates against, and the autofixer derives fixability from, so a doc that the generator produces can never be rejected by the linter.
2. The link graph sees every real edge: wikilinks, `related:` frontmatter refs, and inline repo-path references, so orphan and expansion logic reflect actual connectivity and doc-to-code references become first-class.
3. Semantic retrieval is on by default and incremental: embeddings rebuild per changed chunk by content hash, a cross-encoder reranks the top-k, and any degrade to FTS-only is announced in the response rather than served silently.
4. `created` and `updated` are derived from git history, not hand-typed, killing date rot at the source.
5. `stardust check --fix` remediates every mechanically fixable issue (type injection, git-derived dates, forbidden dashes, off-convention filenames), turning the report into a codemod.
6. Any doc that references code (through `governs:`, `related:`, or an inline path) is flagged when that code has moved since the doc was last touched, surfaced in `check`, in the agent manifest, and in context bundles, making repo docs trustworthy instead of archaeological.
7. CI adopts on a dirty repo without an error wall: a baseline absorbs the existing backlog and the gate fails only on new errors.
8. Wikilinks resolve unambiguously across collections through collection-scoped names.
9. Every note in a bundle carries why it was selected.
10. A bundle declares how far behind HEAD its index is, so the agent knows when to re-verify.

</details>

<details>
<summary><b>Non-goals</b></summary>
<br>

- Indexing or parsing source code semantics. Stardust indexes markdown; doc-to-code binding is a path reference plus git commit-distance, never an AST or a code embedding.
- A network vector database or an external reranker dependency. Vectors stay brute-force-in-Go; the reranker stays an optional local HTTP endpoint that degrades to identity.
- Replacing Obsidian as the human editor or git as the store.
- Auto-rewriting doc prose to match code. Drift is detected and surfaced; the human or agent decides the edit.
- A daemon or always-on server. Watch-mode reindex is at most an opt-in convenience (point 10), not the default posture.
- Changing the JSON-RPC transport or the `--output` contract beyond adding fields.

</details>

<details>
<summary><b>Approach</b></summary>
<br>

The work is three tiers. Tier 1 fixes structural defects that keep the tool mid. Tier 2 makes it trustworthy. Tier 3 is adoption and polish. The drift moat is a Tier 2 capability but is built last, because it depends on the edge extractor (point 2) and the git-date primitives (point 4).

## Tier 1: structural

### 1. One schema, two consumers

The contract moves into the per-collection `config.toml` and every component derives from it.

- `internal/convention/convention.go`: extend `DocCollection` to carry the full ordered field set (or add a `Fields() []collections.Field` builder) so `title, type, status, created, updated` (required) plus `governs, related` (optional `ref`) are declared in one place. `type` is an `enum` of the single collection type; `status` keeps its per-collection enum; `created`/`updated` are `date`; `governs`/`related` are `ref`.
- `internal/cli/init.go:docCollectionConfig`: codegen all seven fields from that declaration instead of hardcoding two. Existing `config.toml` files are left untouched (`writeDocsCollections` already skips existing), so a `rebuild`-style `stardust collections sync` MAY be added to upgrade in place; absent that, the regenerated default plus a one-time migration covers the repo.
- `internal/convention/check.go:checkDocFile`: replace the hardcoded `[]string{"title","type","status","created","updated"}` loop with `collections.LoadOne` for the owning collection followed by `collections.Validate(fm, cfg.Fields)`. The `type`-mismatch and `status`-enum checks fold into `Validate` (it already enforces enums and required). Fall back to the current hardcoded set only when no collection config is present, preserving behavior on un-scaffolded repos.
- `internal/service/checkfix.go`: derive fixability from the schema. A required field that has a derivation rule (`type` from folder, `created`/`updated` from git) is fixable; a required field with no derivation (`title`, `status`) stays report-only. The `fixableKinds` map becomes a function of the loaded schema rather than a static literal.

Outcome: the scaffolder emits exactly what the schema requires, the checker validates against the same schema, and a generated doc is structurally incapable of failing its own linter. See ADR 0014.

### 2. Related and inline refs as first-class edges

Split edge extraction into doc-to-doc edges (graph) and doc-to-code references (drift binding), classified by on-disk resolution, never by guessing.

- `internal/vault/vault.go`: add `ExtractEdges(note Note) []Edge` where `Edge{Target string; Kind string}` and `Kind` is one of `wikilink`, `related`, `inline-path`. Wikilinks come from the existing `wikilinkRe`. `related` targets come from `note.Frontmatter["related"]` via `convention.StringList`. Inline-path candidates come from backtick-delimited inline-code spans whose token matches a `dir/dir/file.ext` shape; a candidate is kept only if it resolves to an existing repo file, which suppresses false positives.
- A resolver classifies each resolved target: a target that is a markdown note in the vault is a graph edge; a target that is a repo file which is not a `.md` note is a code reference. Code references do not become graph nodes; they feed drift (point 6).
- `internal/graph/graph.go`: `Node` gains typed out-edges (carry `Kind` alongside `Target`) so `Orphans`, `BrokenLinks`, `Neighbors`, and `PersonalizedPageRank` operate over the union of wikilink and `related` doc edges. A doc linked only by `related:` stops reading as an orphan and becomes reachable by bundle expansion.
- `internal/convention/check.go:checkRelated` keeps its existence validation; it now also contributes the edges rather than discarding them.

See ADR 0015.

### 3. Turn on the differentiator

Honest framing: vectors are already best-effort-on (the index attempts them whenever Ollama is reachable). The three real gaps are incremental embedding, loud degradation, and an unwired reranker.

- Incremental per-chunk embedding. Persist a per-chunk content hash next to each vector. On reindex, re-embed only chunks whose hash changed, instead of re-embedding every chunk of a note that changed by one character (`index.go:69` today skips at note granularity). This makes embeddings cheap enough to keep on by default.
- Loud degradation. The query and bundle responses, and any future MCP surface, MUST carry a `retrieval_mode` field (`hybrid-semantic` vs `fts-only`) and, when degraded, a one-line reason. The tool stops advertising hybrid while silently serving FTS. `IndexStats.Vectors` already records the run state; surface it on the read path too.
- Cross-encoder rerank. Wire `internal/rerank` into `Query` over the fused top-k when `RerankerURL` is set. It already degrades to identity when unreachable; that degrade is announced under the same `retrieval_mode` discipline.

This stays pure Go and adds no required dependency: with no Ollama and no reranker the tool is an honest FTS engine; with both it is the hybrid-semantic-rerank engine the SPEC promises. See ADR 0016.

## Tier 2: trustworthy

### 4. Git-derive created and updated

- `internal/gitx/gitx.go`: add `FirstCommitDate(ctx, root, path)` (`git log --diff-filter=A --follow --format=%ad --date=short -- <path>`, last line) and `LastCommitDate(ctx, root, paths...)` (`git log -1 --format=%ad --date=short -- <paths>`). Both return a `YYYY-MM-DD` string and an empty string when the path is untracked.
- `internal/service/checkfix.go:fixDocFields`: derive `created` from `FirstCommitDate` and `updated` from `LastCommitDate` when the repo tracks the file, falling back to the current `fileDate` mtime path only for untracked or non-repo cases. A not-yet-committed new doc keeps its scaffold date until its first commit, after which `--fix` reconciles it.

Kills the hand-typed-date error class at the source. See ADR 0017.

### 5. check --fix as a codemod

Extend the existing autofixer (which already fixes `forbidden-dash`, missing `type`, missing `created`/`updated` from mtime):

- Replace mtime dates with the git-derived dates from point 4.
- Add filename remediation for `bad-doc-name`: rename an off-convention file to the convention name (`git mv` to preserve history when tracked), deriving the date prefix from `FirstCommitDate` and the slug from the title for timestamped collections, and the next zero-padded number plus slug for ADRs.
- Keep `title` and `status` report-only (no safe derivation).

The autofixer reports what it changed; re-running `check` proves the remainder. See the plan for the per-issue fixability table.

### 6. Drift detection (the moat, built last)

Generalize the existing commit-distance machinery beyond `governs:`.

- Bind every doc to the code it references through any channel: `governs:` globs (existing), `related:` targets that resolve to code (point 2), and inline code-path refs (point 2).
- For each bound code file, compute commits since the doc's last commit using `gitx.LastCommit` plus `gitx.CommitCountSince`, the same primitives `annotateStaleness` uses today. Remove the `status == "Implemented"` gate for reference-derived bindings; an ADR or research note that points at moved code is stale regardless of an implementation status it never carries. `governs:` bindings MAY keep the Implemented gate for backward compatibility.
- Surface drift in three places: `stardust check` as a `warn` (kind `drift`), the agent manifest (`internal/manifest`), and the context bundle. The message names the doc, the referenced file, and the commit count: "ADR 0001 references `internal/store/daemon.go`, which moved 12 commits since this ADR was last touched; review."

No generic notes tool can do this, because the binding and the commit-distance both require living in the repo. See ADR 0018.

## Tier 3: adoption and polish

### 7. CI baseline and ratchet

- A committed baseline file (`.stardust/baseline.json`) stores a stable fingerprint per known issue (kind plus path plus a normalized detail). `stardust check --ci` reports only issues absent from the baseline and exits non-zero solely on those, so an 81-error repo adopts green and the PR gate catches new rot. `stardust check --update-baseline` snapshots the current set after an intentional change. See ADR 0019.

### 8. Collection-scoped link resolution

- `vault.NormalizeLink` reduces a target to a lowercased basename, so the same slug under `specs/`, `plans/`, and `research/` collides and emits duplicate-name warnings. Add collection-qualified keys (`specs/game-state-backend` or `spec:game-state-backend`) so identical slugs across collections are distinct. Unqualified links keep resolving for backward compatibility, preferring an in-collection target, then a unique global match, then warning on ambiguity. See ADR 0023.

### 9. Provenance in bundle returns

- `service.BundleItem` gains a `Provenance []string` (or a small enum set) recording why the note made the pack: `keyword` (FTS hit), `semantic` (vector neighbor), `link-expansion` (PageRank via a doc edge, carrying the edge `Kind`), or `frontmatter-ref`. The hybrid path already knows FTS vs vector; the PageRank path knows the seed and edge. `packBundle` renders it inline. Cheap to compute, large trust gain for the agent consumer.

### 10. Live freshness

- Stamp every bundle and pack with how far behind HEAD the index is: compute `gitx.CommitCountSince(last_indexed_sha, HEAD)` (the meta cursor `last_indexed_sha` is already written at `index.go:90`) and render "index is N commits behind HEAD" so the agent knows when to re-verify. An opt-in watch-mode reindex MAY follow, but the staleness stamp is the required, boring baseline.

</details>

<details>
<summary><b>Alternatives considered</b></summary>
<br>

- Keep the schema hardcoded in the checker and just sync the three sites by hand. Rejected: the divergence is structural, not a one-time bug; only a single declarative source makes failing docs unrepresentable.
- LLM-extract the doc graph. Rejected by the SPEC already: regex over wikilinks plus frontmatter relations yields more edges, in under two seconds, at zero cost. Point 2 finishes that design rather than reversing it.
- Force vectors on by erroring when Ollama is absent. Rejected: local-first must degrade gracefully; the fix is loud degradation, not a hard dependency.
- Store created/updated in a sidecar database. Rejected: violates files-as-truth and derive-don't-store; git already holds the authoritative timeline.
- Drift via mtime or a stored last-checked timestamp. Rejected: mtimes are unreliable under cloud sync (the reason content-hash exists); commit-distance is the only robust metric and git already provides it.
- One mega-schema shared across all collections. Rejected: each collection has its own status enum and type; per-collection config keeps them honest while sharing the field machinery.
- Embed code files to detect drift semantically. Rejected as a non-goal: commit-distance is exact, cheap, and explainable; a code embedding is none of those and pulls Stardust out of its markdown lane.

</details>

<details>
<summary><b>Risks</b></summary>
<br>

- Inline-path false positives. A backtick span like `foo/bar.txt` that happens to resolve could be mistaken for a code binding. Mitigation: resolve-by-existence plus an extension and source-root allowlist; treat ambiguous tokens as non-edges.
- Schema migration churn. Existing repos have `config.toml` files with only two fields. Mitigation: the checker falls back to the legacy hardcoded set when fields are absent, and a `collections sync` upgrades on demand; no silent breakage.
- Embedding cache invalidation bugs. Per-chunk hashing must invalidate correctly or stale vectors serve. Mitigation: chunk hash derives from the exact embedded text (title plus heading plus body), and `rebuild` always reconstructs from scratch.
- Drift false alarms from formatting-only code commits. A whitespace commit to a referenced file flags drift. Mitigation: commit-distance is a review prompt, not a hard failure; phrase it as "review", and consider a `--drift-since` threshold later.
- Git-date cost on large histories. `--follow` first-commit lookups are O(history) per file. Mitigation: only run in `--fix` and `check`, batch per run, and cache within a single invocation.
- CI baseline staleness. A baseline can hide regressions if never refreshed. Mitigation: `--update-baseline` is explicit and the fingerprint is content-derived, so a fixed-then-reintroduced issue re-fires.

</details>

<details>
<summary><b>Open questions</b></summary>
<br>

1. Inline-path edge scope: restrict to a configured set of source roots (`internal/`, `cmd/`, `src/`, `cli/`) or accept any resolvable repo path? Default: any resolvable non-`.md` repo path, with the source-root list as an optional narrowing filter.
2. Wikilink namespace syntax: path-style `specs/slug` or prefix-style `spec:slug`, each inside double brackets? Default: accept both on read; emit path-style.
3. Drift gate for `governs:` bindings: keep the `Implemented` gate for governs while reference bindings are ungated, or ungate both? Default: keep governs as-is, ungate reference bindings, revisit once noise is measured.
4. Baseline location and format: `.stardust/baseline.json` committed, versus a `# stardust-baseline` block in config. Default: standalone committed JSON.
5. Watch-mode: ship in this work or defer entirely behind the staleness stamp? Default: defer; stamp only.
6. Reranker default: keep `RerankerURL` empty (opt-in) or auto-probe a conventional local port? Default: opt-in, no auto-probe.

</details>

<details>
<summary><b>Verification</b></summary>
<br>

End to end, adversarially:

- Schema: scaffold a doc with `stardust new spec "X"`, then `stardust check` reports zero issues; hand-edit a collection `config.toml` to add a required custom field, re-run, and the checker enforces it without a code change. Delete `type` and confirm `--fix` restores it from the folder; delete `title` and confirm it stays report-only.
- Edges: a doc reachable only through `related:` is not listed by `stardust graph orphans`, and appears in `stardust bundle` expansion. An inline `internal/store/daemon.go` in an ADR body is classified as a code reference, not a graph node.
- Vectors: with Ollama up, `stardust query` returns `retrieval_mode: hybrid-semantic`; stop Ollama and the same query returns `retrieval_mode: fts-only` with a reason, never silently. Touch one heading in a multi-chunk note and confirm only the changed chunk re-embeds.
- Git dates: commit a doc, then `stardust check --fix` sets `created` to the first commit date and `updated` to the last, both matching `git log`.
- Drift: commit an ADR that references `daemon.go`, then push commits to `daemon.go`; `stardust check` emits a `drift` warning naming the file and the commit count, and the bundle and manifest carry it.
- CI: on the current 81-error repo, `stardust check --update-baseline` then `stardust check --ci` exits zero; introduce one new forbidden dash and `--ci` exits non-zero on exactly that issue.
- Namespacing: create the same slug under `specs/` and `plans/`; `stardust check` no longer emits a duplicate-name warning and a `specs/slug` wikilink resolves to the spec.
- Provenance and freshness: `stardust bundle` items each carry a non-empty provenance, and the pack header states the index is N commits behind HEAD.
- Gates: `go build ./...`, `go test ./...`, `go vet ./...`, `gofmt -l .` empty, `make lint` clean, no em or en dashes anywhere.

</details>

<details>
<summary><b>Out of scope</b></summary>
<br>

The MCP server, HTTP/JSON API, full SDK, and Obsidian plugin remain deferred (`CLAUDE.md`); the `retrieval_mode`, provenance, and freshness fields are designed so those later surfaces inherit them for free. Code-semantic indexing, AST analysis, and auto-rewriting doc prose are permanently out of scope. The plugin authoring commands (`/stardust:plan|spec|doc`) are a sibling effort, not this spec.

</details>
