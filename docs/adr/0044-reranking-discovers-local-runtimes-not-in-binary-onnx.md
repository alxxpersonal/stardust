# Reranking discovers a local runtime instead of embedding an ONNX cross-encoder in the binary

SPEC.md section 11 lists "a cross-encoder reranker served in-binary (pure-Go ONNX)" as genuinely-future work. This ADR kills that line as the mechanism and replaces it. The binary never gains an inference runtime. Instead, reranking auto-discovers a cross-encoder served by a local runtime the user already runs, with the fallback chain configured > discovered > none. The pure-Go single-static-binary release property is preserved by construction, and the config story ends strictly simpler than today, where `reranker_url` is a required knob nobody sets.

## Context

Reranking already exists in the tree but is dead weight. `internal/rerank/rerank.go` is an HTTP client that POSTs `{query, documents}` to `<reranker_url>/v1/rerank` and reads back `{results:[{index, relevance_score}]}`, the Jina / Cohere / llama.cpp contract. It degrades gracefully: no URL, fewer than two hits, or any transport or decode error returns the input order unchanged. `internal/service/service.go` calls it after `index.Store.Hybrid` and sets `QueryResult.Reranked` when the order actually changed. The gate is `internal/config/config.go` `RerankerURL` (`reranker_url`), documented as "optional cross-encoder endpoint; empty = disabled" and defaulted to empty. Nobody configures it, so the second-biggest retrieval win in SPEC section 10.2 never fires.

The release property this ADR must not break is concrete. `.goreleaser.yaml` builds with `CGO_ENABLED=0` for darwin and linux on amd64 and arm64: four static tarballs plus a Homebrew formula. SPEC section 9 records the deliberate choice of pure-Go `modernc.org/sqlite` and brute-force cosine in Go specifically to keep this single-binary, trivial-cross-compile property. Any reranking mechanism that demands CGO or a runtime-loaded shared library for the default build forfeits that asset.

The felt pain is the config burden of standing up and naming a separate reranker endpoint. Two questions decide the mechanism: can a cross-encoder run inside the pure-Go binary, and can the already-configured embedder (Ollama) rerank so no new dependency is needed. Both were surveyed against the upstream repositories in July 2026, not from memory.

- **In-binary pure-Go ONNX is a myth for this binary.** `yalue/onnxruntime_go` requires CGO and a runtime `libonnxruntime` shared library per platform, breaking the letter of the `CGO_ENABLED=0` property. `knights-analytics/hugot` does now expose a `crossEncoder` pipeline, but its performant backends (ORT, XLA) are CGO plus `libonnxruntime.so` or XLA libraries plus a Rust `libtokenizers.a`; its one CGO-free backend (GoMLX pure Go) is documented as "for simpler workloads, environments that disallow cgo, and smaller models such as all-MiniLM-L6-v2," best with batches of roughly 32, with the maintainers recommending a C backend for performance. bge-reranker-v2-m3 and Qwen3-Reranker-0.6B (the models SPEC section 10.2 names) sit above that lighter tier, and the pure-Go path additionally pulls the GoMLX framework and still requires distributing the ONNX model weights, which is new binary weight and new artifact plumbing, the opposite of dependency-light. `oramasearch/onnx-go` on Gorgonia does not implement the full ONNX operator set ("most models from the model zoo will not work") and is not viable for a BERT-class cross-encoder. So a lean, dependency-light, model-distribution-free in-binary cross-encoder does not exist today.
- **Ollama cannot rerank today.** Ollama exposes embeddings (`/api/embed`, which the tree already uses) but has no rerank endpoint. PR ollama/ollama#7219, which proposed `/api/rerank`, was closed unmerged on 2025-09-16 as idle; the follow-up PR #11156 and issue #16076 (May 2026) confirm there is still no `/api/rerank`, with the related model-support request #10989 (qwen3-reranker, open since June 2025) drawing no maintainer engagement. The only workarounds are yes/no token log-likelihoods through `/api/generate` (awkward and slow) or query-document embedding cosine, which is the wrong layer of a cross-encoder and adds nothing over the bge-m3 vectors hybrid retrieval already fuses. So "reuse Ollama for a true cross-encoder rerank" does not work now, but it is the obvious future once the endpoint lands.
- **llama.cpp already reranks over the contract the tree already speaks.** `llama-server` serves `/v1/rerank` when launched with `--reranking` (which requires `--embedding` with `pooling=rank`); bge-reranker-v2-m3 (Q4_K_M GGUF) is the reference well-behaved model. `internal/rerank/rerank.go` already targets exactly this endpoint and response shape. The gap is purely that the user must hand-configure its URL.

The through-line: the achievable win is not running the model inside the binary, it is removing the manual endpoint configuration in front of a runtime the user is already likely to run, and making the Ollama-rerank future a config-free auto-activation the day it ships.

## Decision

Ship endpoint-free reranking as local-runtime auto-discovery over the existing HTTP `rerank.Client`. No inference runtime, no CGO, no model weights enter the binary. Five rules govern it.

- **The binary discovers a runtime, never embeds one.** Reranking stays an HTTP call from `internal/rerank` to a cross-encoder served by a local runtime the user or Ollama manages. No CGO backend, no bundled ONNX runtime, no bundled or downloaded model weights. The `CGO_ENABLED=0` four-tarball-plus-brew release property (`.goreleaser.yaml`) is preserved by construction, and a test asserts the rerank and discovery packages import only the standard library net/http path so a future edit cannot silently pull a CGO transitive dependency into the default build.
- **The endpoint is auto-discovered, not hand-configured.** When `reranker_url` is unset, the service probes a short fixed list of well-known local endpoints once at open, sends a two-document canary rerank to each, and caches the first that returns a well-formed `results` array with a relevance score. The probe list is derived from already-configured values (the Ollama base URL) plus the llama.cpp `llama-server` default, so discovery introduces no new required config and mirrors the cheap `embed.Available` probe philosophy. The probe runs only when no URL is configured, is cached for the service lifetime, and never blocks or fails a query.
- **Fallback chain: configured > discovered > none.** An explicit `reranker_url` always wins as override and escape hatch, including a `none` sentinel that hard-disables probing for users who do not want localhost discovery. Absent that, a discovered runtime is used. Absent that, reranking is off and `index.Store.Hybrid` order is returned unchanged. The active source is announced on the result (extending the ADR 0016 loud-degradation rule to the rerank stage) so a consumer never mistakes "no runtime found" for "reranker ran and kept the order."
- **Reuse Ollama the instant it can rerank, with zero code change.** Discovery probes the already-configured Ollama host for a rerank endpoint. Today Ollama has none, so that probe yields nothing at the cost of one cached 404. The day Ollama merges `/api/rerank`, discovery over the same host the semantic user already runs for embeddings lights reranking up with zero user action and zero new dependency. The forward-compatible seam is the whole reason discovery beats a bare default URL: the reuse-Ollama payoff is pre-wired, not a later migration.
- **In-binary ONNX is rejected on the evidence, and the revisit trigger is narrow.** The pure-Go ONNX line in SPEC section 11 is struck as the mechanism. It is revisited only if a pure-Go ONNX path (concretely, hugot's `crossEncoder` pipeline on a matured GoMLX backend) can run a bge-reranker-v2-m3-class cross-encoder at retrieval-grade quality and latency, AND a model-distribution story exists that neither fattens the static binary nor adds a required config or cache knob. Until both bars clear, discovery is the answer and the Ollama-rerank future is already absorbed by rule four, so no revisit is needed for the reuse-Ollama case.

## Consequences

- The config story ends strictly simpler. Today reranking is off unless the user sets `reranker_url`, which nobody does, so the feature is dead. After, reranking auto-enables whenever a known local runtime is up, `reranker_url` becomes an optional override rather than a required knob, and the common case (a user running `llama-server`, or later any Ollama with rerank) is zero-config. No new required field is added; the net required surface shrinks to zero.
- The release property is untouched. Discovery and reranking are pure net/http plus JSON, so `CGO_ENABLED=0 go build ./...` stays green and goreleaser still emits four static tarballs and a working brew formula. The import-boundary test makes that regression-proof.
- Reranking stops being latent. A cross-encoder over the top ~50-100 hits down to the returned k is SPEC section 10.2's second-biggest retrieval win; discovery is what makes it fire without a config ritual, so the win is realized rather than documented.
- Value is real but conditional, and the docs say so. Discovery finds a runtime only when one is running. For a user with nothing rerank-capable up, the surface stays off (announced), exactly as today, minus the dead knob. The payoff compounds with the Ollama-rerank future, which discovery captures for free.
- One new small package (the probe plus per-runtime contract adapters), no new heavyweight dependency, no new index table, no CGO, no model artifact. The blast radius is a discovery step in front of an HTTP client that already exists and already degrades safely.

## Alternatives considered

- **Embed a pure-Go ONNX cross-encoder in the binary (the struck SPEC line).** Rejected. No lean pure-Go path runs a bge-reranker-v2-m3-class model today: `yalue/onnxruntime_go` is CGO plus a shared library, hugot's performant backends are CGO, and hugot's CGO-free GoMLX backend targets "simpler workloads / smaller models" below the reranker tier while pulling a heavy framework and still needing model-weight distribution. This breaks either the letter (CGO, shared lib) or the spirit (heavy dependency, fat binary, new artifact plumbing) of the single-static-binary release asset.
- **Bundle or download the ONNX model weights and run them via a CGO backend gated behind a build tag.** Rejected for the default distribution. A build-tagged CGO reranker means the shipped brew and tarball binaries (the ones users actually get) cannot rerank, so it does not reduce the config burden for the audience that has it, and it fractures the "one binary, same behavior everywhere" property.
- **Reuse Ollama now via the embedding-similarity or logprobs workaround.** Rejected. Ollama has no rerank endpoint (PR #7219 closed unmerged, no `/api/rerank` as of mid-2026). Query-document embedding cosine is the wrong layer of a cross-encoder and adds nothing over the bge-m3 vectors hybrid already fuses; the `/api/generate` logprobs trick is slow and awkward. Discovery instead pre-wires the real endpoint so the genuine reuse-Ollama win activates the moment upstream ships it.
- **onnx-go on Gorgonia.** Rejected. Incomplete ONNX operator coverage; most model-zoo models, including BERT-class cross-encoders, will not run. Not a viable inference path.
- **Keep the required `reranker_url` and just document it louder.** Rejected. The problem is that a required endpoint knob is dead weight; louder docs do not change that nobody stands up and names a separate server. Discovery removes the required step entirely.
- **Ship nothing, defer the whole feature.** Rejected. A defer would leave a required-but-dead knob in place and would not satisfy the standing requirement that the config story end simpler. Discovery is achievable now, CGO-free, and dependency-light, so the honest move is to build the achievable win and strike only the unachievable in-binary mechanism.
