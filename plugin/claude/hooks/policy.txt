<stardust-context-policy>
  <reason>
  This workspace stores its plans, specs, decisions, and verification state as markdown
  that stardust indexes for hybrid keyword and semantic retrieval. The index is the
  freshest and most complete view of what has been decided and what is in flight. Routing
  context through stardust returns the right notes, with their links and file paths, in one
  call. Reaching for ad hoc grep or opening files by guess skips that index, so the likely
  result is stale context: an active plan goes unseen, a settled decision gets contradicted,
  and work already finished gets redone. One query or bundle call prevents that.
  </reason>

  <routing>
  - For context on a task, ticket, or feature, run one bundle: stardust bundle "<task>".
    It returns a budgeted, link-expanded pack of the most relevant notes with paths. Read
    that before opening files by hand.
  - To locate a prior decision, spec, plan, or note, run stardust query "<text>" first.
  - When the stardust MCP server is connected, call the bundle and query tools instead of
    shelling out; they return the same results in-process.
  - Use grep, the editor, and the language server for source code. stardust indexes
    markdown docs and decisions, not code. grep finds a symbol; stardust finds the plan
    that explains it.
  - After a decision is settled or a unit of work lands, record it with
    stardust remember "<fact>" so the next session boots already aware of it.
  </routing>
